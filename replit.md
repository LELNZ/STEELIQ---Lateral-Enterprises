# SteelIQ – Lateral Enterprises

## Overview
SteelIQ is a professional quotation and estimating platform designed for the window and door industry. It automates and optimizes the quotation workflow, providing features like item configuration, real-time visual representation, accurate pricing, and efficient document generation. The platform aims to deliver live SVG technical drawings, comprehensive estimate and quote lifecycle management, detailed pricing breakdowns, and a flexible template-driven PDF engine. It includes tools for managing organizational and division-specific settings, dynamic specification displays, and a project-centric CRM workflow. SteelIQ's vision is to streamline operations, enhance accuracy, and provide a competitive edge for businesses in the window and door sector.

## System Architecture
**Frontend**: Built with React and TypeScript, utilizing Shadcn UI for a responsive, mobile-first user experience.
**Backend**: Developed using Express.js, with a PostgreSQL database managed via Drizzle ORM and node-postgres.
**Drawing Engine**: Employs SVG for real-time technical drawing rendering and supports PNG export.
**State Management**: React state manages UI components, while TanStack Query handles API data fetching and caching. Global settings are managed via React Context and localStorage.
**Routing**: Client-side navigation is handled by Wouter, with a centralized route helper (`client/src/lib/routes.ts`) to ensure consistency across the application.
**Export Capabilities**: Supports SVG to PNG conversion and multi-page, vector-text selectable PDF generation using jsPDF, with specialized features for Subcontractor PDFs (Document Purpose selector, item filtering, raked-fixed dimension rendering). PDF generation includes performance optimizations like yield points and progress callbacks. Customer Quote Previews render as paginated A4-style pages.
**Storage**: Item photos and drawing PNGs are stored in PostgreSQL (`bytea`), with an in-memory cache and filesystem mirror. Division logos are stored as base64 data URLs in PostgreSQL.
**Drawing Artifact Lifecycle**: Drawing canonical truth is stored in `specValues` within `EstimateSnapshot` revisions. The snapshot now includes all 18 fields required for exact DrawingCanvas regeneration (including `customColumns`, `entranceDoorRows`, `entranceSidelightRows`, `entranceSidelightLeftRows`, `hingeDoorRows`, `frenchDoorLeftRows`, `frenchDoorRightRows`, `panelRows`, `rakedSplitEnabled`, `rakedSplitPosition`, `sidelightEnabled/Side/Width`, `bifoldLeftCount`, `centerWidth`, `doorSplit/Height`, `halfSolid`). Preview is read-only — it shows stored images or DrawingCanvas SVG fallback but does NOT write on read. Drawing repair/regeneration is **server-side only** via `POST /api/drawing-images/regenerate` (admin/owner role required). The server uses `renderToStaticMarkup` + `sharp` to render DrawingCanvas SVG → PNG without any browser DOM. The client "Repair Drawings" button calls this server endpoint — no browser-side `createRoot` or DOM rendering. `classifySnapshotDrawingSupport()` categorizes each item as exact/approximate/basic based on available snapshot data. PUT `/api/drawing-images/:key` and POST `/api/drawing-images/check-missing` require admin/owner role (403 for non-privileged).
**Multi-Division Architecture**: Supports organizational and division-specific settings for enhanced flexibility. Each division has a `domainType` (joinery/engineering/laser/general) that drives conditional UI behavior. Users support multi-division assignment via `divisionCodes` text array (backward-compatible with legacy `divisionCode` string). Division access filtering uses centralized `getUserDivisions()`/`isUserAllDivision()`/`userCanAccessDivision()` helpers. Constants `DOMAIN_TYPES` and `DIVISION_DOMAIN_MAP` defined in `shared/schema.ts`.
**Domain-Discriminated Contracts (Platform Prep 0B)**: The quote/item/snapshot/render pipeline is domain-aware. `resolveQuoteDomainType(divisionCode)` resolves a division to its domain type. The preview-data API returns `domainType` which flows through `QuoteDocumentModel` → `QuoteRenderModel` → Preview/PDF. Item contracts use discriminated types: `JoineryItemPayload` (= existing `QuoteItem`), `LaserItemPayload`, `EngineeringItemPayload` as domain-specific shapes wrapped in `DomainItemEnvelope`. Snapshot items have corresponding `JoinerySnapshotItem`, `LaserSnapshotItemBase`, `EngineeringSnapshotItemBase` with `DomainSnapshotItemEnvelope`. The pricing engine (`PRICING_ENGINE_DOMAIN = "joinery"`) and drawing regenerator (`DRAWING_ENGINE_DOMAIN = "joinery"`, `isDomainDrawingSupported()`) are explicitly scoped to Joinery. Non-Joinery drawing regeneration requests are safely skipped.
**Spec Dictionary System**: Provides configurable entries for dynamic display and override of specifications.
**Laser Foundation (Phase 1 + 1A + Phase 2 Pricing)**: LL (Lateral Laser) division supports a complete quote workflow: manual item entry via dedicated Laser Quote Builder page (`/laser-quote/new`, `/laser-quote/:id`), snapshot persistence with `laserItems[]` array in `EstimateSnapshot`, preview and PDF rendering branched on `domainType === "laser"`. Laser items map to `QuoteDocumentItem` with material specs in `resolvedSpecs` (materialType, materialGrade, thickness, dimensions, finish). Joinery-specific features (drawings, GOS, cat door, pane glass specs, opening direction) are suppressed for Laser items. The `POST /api/quotes/:id/revisions` route supports direct revision creation with division access control. Sidebar navigation links to the Laser builder, and the quote detail page provides an "Open Laser Builder" link for LL quotes. **Phase 1A additions**: `ll_sheet_materials` table (supplier-linked sheet material master with materialFamily, grade, finish, thickness, sheet dimensions, pricing); CRUD API at `/api/ll-sheet-materials`; "Sheet Materials (LL)" admin tab in library page; Laser builder material selects are now library-driven (cascading family→grade→finish→thickness from active sheet materials); library scope correction scopes LJ-specific entries (glass, frame, hardware, labour, etc.) with `divisionScope=LJ`, preserving genuinely shared entries (installation/delivery/removal/waste rates). **Real supplier seed data**: 257 rows from Macdonald Steel (Lateral Engineering Pricelist Nov 2025: Mild Steel HA300/G250/Cold Rolled, Galvanised, Corten, Stainless 304L/316L) and Wakefield Metals (Price List 42A2: Aluminium 5005/5052/5083/6061, Stainless 304/316/430). Seed function auto-purges demo data on startup. Seed data defined in `server/ll-seed-data.ts`. **Phase 2 Pricing Engine**: LL pricing engine lives in `client/src/lib/ll-pricing.ts` — entirely separate from LJ pricing. Three-layer separation: (A) Procurement Truth (supplier-linked `ll_sheet_materials` row with `llSheetMaterialId`), (B) Pricing Engine Truth (`computeLLPricing()` derives material cost via rectangular area basis with configurable utilisation factor, process cost from cut length + pierce count, setup/handling from operator-entered minutes, and commercial markup), (C) Customer Output Truth (only sell result flows to preview/PDF via existing `unitPrice` field). Items persist `llSheetMaterialId`, `cutLengthMm`, `pierceCount`, `setupMinutes`, `handlingMinutes`, `markupPercent`, `utilisationFactor` on `LaserQuoteItem`. Snapshots additionally store denormalized material truth (`supplierName`, `sheetLength`, `sheetWidth`, `pricePerSheetExGst`) and full pricing breakdown (`materialCostTotal`, `processCostTotal`, `setupHandlingCost`, `internalCostSubtotal`, `markupAmount`, `sellTotal`) for audit immutability. Internal pricing breakdown is visible in the builder via expandable panels per item and inline in the Add/Edit dialog. Default rates: $0.012/mm cut, $0.50/pierce, $95/hr shop rate, 75% utilisation factor, 35% markup. Minimum material charge: $25. The manual `unitPrice` input field has been replaced — `unitPrice` is now derived from the pricing engine (`sellTotal / quantity`).
**Library Category Ownership Model**: The library shell uses an explicit `CATEGORY_OWNERSHIP` config (in `client/src/pages/library.tsx`) that declares each category's owner, shared status, and justification. LJ-owned: Direct Materials, Manufacturing Labour, Glass, Frame Types, Frame Colors, Hardware, Liner Types, Wanz Bar, Profile Roles, Site Costs. LL-owned: Mild Steel, Aluminium, Stainless Steel, Galvanised Steel, Corten (split from single "Sheet Materials" into per-family categories). LE-owned: none yet. Shared: none. Each division scope shows only its owned categories. In "All" view, tab labels are prefixed with division owner (e.g. "LJ – Glass", "LL – Mild Steel") for enterprise clarity. LE sees an empty state message. LL family categories include quick filters for supplier, grade, finish, and thickness.
**Quote Management**: Features a comprehensive lifecycle (Draft, Review, Sent, Accepted/Declined, Archived) with atomic sequential numbering, immutable revision history, and server-side status enforcement. Each quote includes an `EstimateSnapshot` for immutable revision data. Estimate↔quote link truth: linked quotes are determined by `source_job_id` on the `quotes` table (no stale pointer field). Quote deletion invalidates `/api/jobs` cache to clear linked-quote badges. Revision mode (`POST /api/quotes` with `mode: "revision"`) returns 404 if no valid quote exists (soft-deleted excluded via `deleted_at IS NULL`), preventing silent new-quote creation. Revision creation updates `quotes.customer` from the request to keep preview/PDF customer-facing truth current. Revision targeting is deterministic (`ORDER BY created_at ASC LIMIT 1`).
**Pricing System**: Calculates material, labor, glass, liner, and handle costs, providing net cost, sale price, and margin. Supports manual price override, removal/rubbish fees, and markup application. Specific functionalities include "Glaze on Site" (GOS) charges as a sell-side revenue component, "Cat Door" flagging, and "Outsourced Item Commercial Override" for separate cost/sell calculations for outsourced items. Installation and removal pricing support various bases (per_item, per_m2, per_lm). Geometry-aware pricing utilizes grouped geometry metrics.
**Pane-Level Glass Selection**: Items with Height from Floor ≤ 800mm and multiple panes expose per-pane glass selectors (IGU Type, Glass Type, Glass Thickness). Data stored in `paneGlassSpecs` array on the item schema and carried through the snapshot → document → renderer → preview/PDF pipeline.
**Configuration & Drawing**: Dedicated database tables for frame configurations, profiles, accessories, and labor. Features auto-detection of configurations, dynamic opening indicators, and support for various window/door types (e.g., French windows, stacker doors, entrance doors with auto-calculated sidelights, raked/triangular fixed windows, bay windows). Bay windows utilize `bayAngle` and `bayDepth` for visual approximation in drawings. Opening directions are structured and category-aware. Raked fixed windows support `rakedLeftHeight`/`rakedRightHeight` for trapezoidal area calculation and perimeter overrides. Geometry-aware auto-configuration uses a `geometryClass` field and adds transom profiles.
**Master Library Systems**: Centralized libraries for direct materials, manufacturing labor, installation labor, and delivery methods. The Manufacturing Labour Library includes admin controls for Active/Inactive status and default assembly designation. The Glass Pricing Library supports multiple IGU families with dynamic thickness matrices. The "Reset to Defaults" and "Seed from Configurations" buttons have been removed from the Item Library product UI to eliminate accidental reseed risk. Backend seed routes (`POST /api/library/seed`, `POST /api/frame-types/seed-configurations`, `POST /api/library/seed-direct-materials`) remain as internal bootstrap-only code used during initial server startup when the library is empty — they are not reachable from the product UI.
**CRM Workflow (Projects-First)**: Project-centric CRM with automatic customer linking from quotes, project creation linked to quotes, and sub-query endpoints.
**Invoice Management**: Manages invoices with `invoice_lines` for detailed line items, full CRUD operations, and automated status demotion. The invoice detail page provides a dense operational layout with status badges and Xero integration status.
**Xero Integration**: Supports Xero payment visibility and status readback (`xeroStatus`, `xeroAmountPaid`, `xeroAmountDue`). Includes a "Sync from Xero" button and a "Xero Link Reset / Reissue Workflow."
**Job/Quote/Invoice Linkage**: Invoice detail and list pages resolve job name and variation title through linked records, and invoice creation auto-populates `reference` from the source job name.
**Variations Commercial Model**: Manages project-level variations through a defined lifecycle.
**Retention Commercial Model**: Supports optional percentage-based retention.
**UI/UX**: Features responsive tables, polished spec row formatting in PDFs, project dashboard guidance, customer relink safeguards, and an enterprise app shell with workflow-domain grouped navigation. An accepted quote workflow progress panel guides users through project, job, and invoice creation. The estimate list provides a three-signal commercial control surface for Estimate Status, Quote status, and Revision.
**Governance and Data Integrity**: Includes features for environment clarity, test data governance, and shielding of demo/test records from standard users.
**Pane Data Integrity (Phase 2)**: Shared pane integrity helpers in `shared/pane-integrity.ts` provide validation, classification (all-valid/some-unresolved/all-unresolved/no-overrides), and fallback explanation for pane glass specs. The quote-builder UI shows per-pane validation indicators (green check/red X/amber warning), an integrity summary banner, detailed fallback pricing explanations, save-time validation blocking incomplete specs, and "Clear All"/"Clear" correction controls. Items with invalid pane specs display a "Pane Issue" badge in both desktop table and mobile card list views.
