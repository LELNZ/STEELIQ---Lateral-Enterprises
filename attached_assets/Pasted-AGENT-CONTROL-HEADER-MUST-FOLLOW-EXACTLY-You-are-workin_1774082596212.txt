==================================================
WORKSTREAM A — RETENTION COMMERCIAL MODEL
==================================================

OBJECTIVE
Model retention as a real withheld commercial value, not as an accidental leftover amount.

Required behavior:

1. Retention on accepted commercial scope
- Retention must be configurable on the accepted commercial scope
- Support either:
  - retention percentage
  - or retention amount
- If both are currently too risky, choose the safest single model and explain why
- Preferred:
  - retentionPercentage
  - derived retentionHeldValue

2. Retention held visibility
- Billing summaries must clearly show:
  - base contract value
  - approved variations
  - total invoiceable before retention
  - retention held
  - invoiceable before retention release
  - total invoiced
  - remaining
- Users must understand that retention is withheld, not forgotten

3. Retention impact on standard invoices
- Deposit/progress/final invoices must not accidentally consume value reserved as retention release
- Final invoice logic should account for retention being withheld unless/until released

4. Retention state
At minimum, track:
- retention configured
- retention held value
- retention released value
- retention remaining to release

If status is useful, implement it carefully.
Do not overbuild.

==================================================
WORKSTREAM B — RETENTION RELEASE INVOICE FLOW
==================================================

OBJECTIVE
Allow controlled billing of retained value.

Required behavior:

1. Retention release invoice type
- Implement a dedicated retention release invoice flow
- It must bill only from retention held / remaining to release
- It must not exceed unreleased retention value

2. Retention release calculation
- If retention held = $X and retention already released = $Y
- remaining retention release allowance = $X - $Y
- retention release invoice must be capped to that amount

3. Invoice inheritance
Retention release invoices must inherit:
- customer
- project
- source quote
- linked job where relevant
- division
- normal Xero/invoice pipeline compatibility

4. Remaining-to-invoice logic
- Remaining contract billing and remaining retention release must be separately understandable
- Users must not confuse:
  - standard uninvoiced balance
  - retention still withheld
  - retention available to release

WORKSTREAM C — CLOSEOUT BILLING SUMMARY LOGIC
==================================================

OBJECTIVE
Make the commercial end-state of a project/quote clearer.

Required behavior:

1. Billing summary must clearly distinguish:
- Base contract
- Approved variations
- Total contract including variations
- Retention held
- Standard invoiced total
- Retention released total
- Total invoiced overall
- Remaining standard billing
- Remaining retention release

2. Final invoice logic
- Final invoice should auto-suggest the remaining standard billing balance
- It should NOT silently consume retention-held value unless explicitly using retention release flow

3. Closeout clarity
- On quote/project commercial surfaces, staff should be able to tell:
  - whether contract billing is fully complete apart from retention
  - whether retention is still outstanding
  - whether the commercial scope is fully billed including retention

Do not build a giant closeout module in this phase.
Just make billing state understandable and operationally safe.
