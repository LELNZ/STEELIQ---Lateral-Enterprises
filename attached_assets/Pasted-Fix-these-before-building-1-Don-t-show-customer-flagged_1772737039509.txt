2) Backward-compat: don’t “synthesize photos[] if photo exists as a non-base64 key” (too risky)

Right now discovery says jobItems.photo stores base64 strings in DB. Your plan suggests:

“if photos is null but photo exists as a non-base64 key…”

That’s dangerous because you need a reliable test of “key vs base64”.

Best-practice compatibility rule:

If photo starts with data:image/ (or is very long), treat as legacy base64 and do not try to convert automatically.

If photo matches your new key regex (uuid.jpg), treat it as a key.

Even better: write a migration helper in the client:

If legacy base64 exists, show it in UI, but require the user to re-add/upload if they want it preserved as a file key. (Optional “Convert legacy photo” can come later.)

Auto-converting base64 to filesystem on read is possible, but it’s more work and easy to get wrong—don’t do it unless you explicitly choose to.
