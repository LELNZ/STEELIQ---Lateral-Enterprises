PART 1 — CANONICAL INVOICE SOURCE LOCK
==================================================

Tighten invoice creation so the system has one safe commercial source rule.

Required rule for this phase:
- invoice creation must originate from accepted quote context
- if quote-based invoice creation is used, the system must derive source truth from the accepted quote, not caller overrides

Required behavior:
When POST /api/invoices is called with quoteId:
- quote must exist
- quote must be accepted
- quoteRevisionId must be derived from quote.acceptedRevisionId
- customerId must be derived from quote.customerId
- projectId must be derived from quote.projectId
- divisionCode must be derived from quote.divisionId

Do NOT allow caller-supplied values to override accepted source truth for these fields when quoteId is present.

If quote.acceptedRevisionId is missing, fail safely and clearly rather than creating an ambiguous invoice.

If there are any other invoice creation paths in UI or backend, align them to this canonical rule or report them truthfully.

PART 2 — PUSHED INVOICE EDIT SAFETY
==================================================

Define and enforce safe rules for invoices after Xero push.

Required rule:
Once an invoice has been pushed to Xero, commercial editability must be tightly controlled.

For this phase, implement the safest minimal rule:
- pushed_to_xero_draft and approved invoices must not allow casual editing of core commercial fields
- core commercial fields include at minimum:
  - type
  - quoteId / quoteRevisionId
  - customerId
  - projectId
  - divisionCode
  - depositType
  - depositPercentage
  - amountExclGst
  - gstAmount
  - amountInclGst
  - description
- notes may remain editable only if that does not create accounting ambiguity; if unsure, lock notes too and report it

If a pushed invoice must be changed:
- user must return it to draft through the controlled path
- user must see the warning that the Xero invoice must be deleted before replacement/reissue
- do not silently overwrite pushed accounting-linked records

Do NOT implement a full replacement engine yet.
Do NOT allow broad mutation of pushed invoices.
