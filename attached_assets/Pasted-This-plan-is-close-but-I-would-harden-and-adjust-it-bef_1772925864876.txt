What I would change
1. Do not assume the estimate entity is always called job

Your plan uses jobId and /api/jobs/:id/....

That may be correct in the current codebase, but it should be treated as a codebase discovery item, not locked in at planning level.

Why this matters:

business language in the docs is estimate/quote/site visit

if the DB/API happens to use job, fine

but the implementation should preserve the domain distinction between:

estimate record

quote record

future QuoteDocument output

Hardened change:
Use “linked estimate/job entity” in the plan, and let the agent confirm the actual storage key and route naming before coding.

2. Add lifecycleAction rules, not just status

Right now the plan focuses on status fields only.

That is not enough for safe behavior, because the key behavior is cascade handling when the parent estimate is archived/deleted.

I would explicitly require:

archive linked quotes

soft-delete linked quotes

hard-delete linked quotes

keep linked quotes but mark orphaned

This logic belongs in a service-layer lifecycle helper, not scattered in routes.

Hardened change:
Add a dedicated quote lifecycle service/module, even if small.

3. Avoid using hard delete as a normal first-class route unless explicitly admin/dev

For production/business safety, I would be careful with:

DELETE /api/quotes/:id as permanent delete

That is acceptable only if:

clearly explicit

role-gated if roles exist

confirmed in UI

not the default path

Given the docs show quotes are evolving toward formal customer-facing output, archive-first is the safer norm. 

STEELIQ_CURRENT_STATE

Hardened change:
Prefer:

archive quote

soft delete quote

hard delete quote only via explicit destructive action, with clear separation from normal delete UX

4. Do not overload status="deleted" if soft-delete timestamps already exist

If you add both:

status = deleted

deletedAt

that is workable, but it creates dual truth unless you define strict rules.

I recommend one of these approaches:

Preferred

keep business-visible status values such as draft, generated, archived

use deletedAt for soft deletion

excluded-by-default records are those with deletedAt != null

Acceptable fallback

keep status="deleted" only if the existing status machine already depends on it heavily

Because you do not want page filters, lifecycle rules, and future rendering logic all branching on two independent delete signals.

Hardened change:
Make one canonical delete rule:

soft deleted iff deletedAt != null

archived iff archivedAt != null or status="archived" depending on existing schema conventions

choose one canonical source of truth and document it

5. Add quote source/version metadata now, even if minimal

Since QuoteDocument/Renderer is next, I would future-proof lightly by checking whether quotes need:

sourceEstimateVersion

quoteRevisionNumber

generatedFromEstimateAt

Not all must be implemented now, but the plan should at least avoid blocking these later.

The docs explicitly say QuoteDocument is expected to support structured pricing, grouped specs, drawings, and photos, and current preview does not yet fully render all fields. 

STEELIQ_QUOTE_DOCUMENT_MODEL

Hardened change:
At minimum, leave room in the model and do not make assumptions that one estimate can only ever have one quote.

6. T002 should explicitly support multiple quotes per estimate

Your plan already improves getQuoteByJobId to getQuotesByJobId, which is correct.

That is important and should stay, because:

a single estimate may later produce multiple quote revisions

renovation/new build/tender variants may eventually diverge

archived prior versions may need retention

So this part is strong and should remain.

7. Add archive support to the estimate side too, not only delete cascade

Your plan has delete cascade UI, but the business requirement also mentioned:

delete or archive estimate

ask whether the quote should be deleted or archived with it

So the frontend work should cover:

estimate delete flow

estimate archive flow

Hardened change:
Add a sibling task for archive-cascade UI, not just delete-cascade UI.

8. Quotes page tabs are good, but “Active” needs a precise rule

You wrote:

Active = all non-archived, non-deleted quotes

That is good, but I would define it more precisely as:

Active

quotes with deletedAt == null

not archived

regardless of quote type

Then:

Renovations = active renovation quotes

New Builds = active new_build quotes

Tenders = active tender quotes

Archived = archived quotes only

This prevents confusion where archived renovation quotes appear in Renovations tab.

9. Orphan detection should be computed, not persisted

Do not store isOrphaned in the DB unless absolutely necessary.

Best practice:

compute orphan state in the API response by checking if linked estimate/job still exists

return it as derived response metadata

That avoids stale flags.

Hardened change:
API enriches quote list payload with:

isOrphaned

linkedEstimateExists

10. The dev clear utility should be dev-only and hard-blocked in production

This must be explicit.

I would require:

route only available when NODE_ENV !== "production" or an explicit dev flag is true

log usage

return count deleted
