CURRENT-STATE CONSTRAINTS
- Existing app is a centralized TypeScript Express + React system.
- Existing jobs, quotes, accepted quote conversion, invoices, admin/library patterns, customers, contacts, projects, and op-jobs already exist.
- There is already Xero push logic and current invoice push failure occurs when no customer name or Xero contact ID is available.
- We must preserve existing app structure and improve the customer/project/job workflow so Xero push can be tested reliably.

PRIMARY GOALS
1. Ensure estimate -> quote automatically preserves and links customer correctly.
2. Ensure customer is auto-created or auto-linked when estimate becomes quote.
3. Keep ability to unlink/reassign customer later with safeguards.
4. Keep Projects as a first-class umbrella record.
5. Make Project creation happen before Job creation in the accepted-quote workflow.
6. Ensure project is created using source estimate/job name and address by default.
7. Ensure Job is created only after Project exists, and links to quote + project + customer.
8. Ensure invoices inherit valid customer/project context so Xero push can be tested successfully.
9. Improve quote detail UX:
   - quote cards collapsible
   - default collapsed
   - lifecycle section moved near bottom
10. Add a real Project dashboard page where quotes, jobs, and invoices are accessible inline without forcing navigation away.

DEFINITIONS TO LOCK
- Customer = who the work is for
- Project = umbrella site/contract/grouping for related scopes
- Quote = commercial proposal
- Job = internal execution record created after accepted quote and linked to project
- Invoice = billing record linked to quote/customer/project

TARGET WORKFLOW
Estimate -> Quote -> Accepted Quote -> Create Project -> Create Job -> Create Invoice(s)

IMPORTANT BUSINESS RULES
- Customer should be effectively required by the time a quote exists.
- Project should remain optional at quote stage, but preferred/strongly guided once quote is accepted.
- Job should not be created before project in the preferred accepted-quote workflow.
- Invoice can be created only if customer linkage is valid enough for Xero payload generation.
- “Ready for Xero” must not be allowed if required customer/Xero linkage is missing.

PHASE 1 - CUSTOMER LINKING + XERO-READY FOUNDATION
==================================================

OBJECTIVE
Fix the root data-linking problem so estimate -> quote creates a real customer relationship and invoices can be tested for Xero push.

IMPLEMENT
1. Estimate -> Quote customer carry-forward logic
   - When estimate converts to quote:
     a. If estimate already has customerId, carry it forward automatically.
     b. If no customerId exists, attempt to find an existing customer by matching:
        - exact name first
        - then email if present
        - then phone if present
     c. If no safe match exists, auto-create a new customer record from estimate customer details.
     d. Link quote.customerId automatically.
     e. Preserve visible customer snapshot fields on quote for historical document consistency.

2. Customer unlink/relink flexibility
   - On quote detail, provide:
     - Linked Customer card
     - Change customer
     - Unlink customer
   - If downstream records already exist (project/job/invoice), show confirmation warning before unlink/reassign.
   - Do not silently rewrite downstream finance records already pushed to Xero.

3. Xero readiness validation
   - Add a validation layer before invoice status becomes "ready_for_xero".
   - Block transition to ready_for_xero unless:
     - invoice has linked customerId OR valid customer name snapshot
     - and customer has enough information to build Xero payload
   - Prefer valid linked customer record.
   - If system requires xeroContactId in your workflow, clearly show that requirement.
   - If name-only mode is acceptable for payload mapping, allow it.
   - Error messaging must be specific and actionable.

4. Invoice push error improvements
   - Improve push-to-Xero error UI so user sees:
     - why it failed
     - whether customer is missing
     - whether Xero contact ID is missing
     - direct action buttons/links:
       - Open quote
       - Open customer
       - Link customer
   - Preserve existing push behavior otherwise.

5. Quote detail page UX cleanup
   - All quote cards/sections should be collapsible.
   - Default state = collapsed for all non-critical sections.
   - Keep quote summary/header open.
   - Move Lifecycle section near bottom of page.

PHASE 1 UI REQUIREMENTS
- Quote detail page:
  - collapsible cards
  - linked customer section
  - clear linked/unlinked state
  - customer change/unlink actions
  - lifecycle lower on page
- Invoice detail/list:
  - clear Xero readiness validation state
  - clear reason why push is blocked
- Customer selection UX:
  - show whether selected customer is existing linked record or auto-created record

PHASE 1 ACCEPTANCE CRITERIA
- Converting estimate to quote always results in a usable customer linkage path.
- Quote displays linked customer correctly.
- User can unlink/reassign customer with safeguards.
- Invoice cannot enter ready_for_xero if customer linkage is invalid.
- Xero push testing can proceed on at least one invoice with a valid customer.
- Quote cards are collapsible and default collapsed.
- Lifecycle is moved lower on the quote page.

PHASE 2 - PROJECT-FIRST ACCEPTED QUOTE WORKFLOW
==================================================

OBJECTIVE
Make Project the umbrella before Job creation and align accepted quote flow to real business use.

IMPLEMENT
1. Accepted Quote action redesign
   - On accepted quote, primary next action should be:
     - Create Project
   - Secondary actions:
     - Create Deposit Invoice
     - Create Progress Invoice
     - Create Final Invoice
     - Create Job (disabled or discouraged until project exists, depending on logic below)

2. Project creation from accepted quote
   - Add "Create Project from Quote" action.
   - Prefill project fields from source quote/estimate:
     - customerId from quote
     - name from source estimate/job/site name if available
     - address from source estimate/job address
     - division where useful
     - description/notes from quote context if applicable
   - On save:
     - link project back to quote
     - keep customer link
     - show project on quote page immediately

3. Project-first job logic
   - Preferred rule:
     - Create Job button appears only after project exists for that accepted quote
     OR
     - if clicked early, system prompts user to create/select project first
   - Job creation must carry:
     - sourceQuoteId
     - acceptedRevisionId
     - customerId
     - projectId
     - title from quote/project
     - address/site context

4. Quote/project linkage visibility
   - Quote detail page should clearly show:
     - linked customer
     - linked project
     - source estimate/job relationship
   - If no project linked yet:
     - show empty-state prompt to create project

5. Invoice inheritance
   - When invoice is created from accepted quote:
     - inherit customerId
     - inherit projectId if project exists
     - inherit accepted revision
     - inherit division
   - If project exists later and invoice is still draft/not pushed, allow controlled relink.

PHASE 2 UI REQUIREMENTS
- Accepted quote action area redesigned
- Create Project flow from accepted quote
- Clear project card on quote page
- Job creation flow tied to project
- Clear linked source relationship display

PHASE 2 ACCEPTANCE CRITERIA
- Accepted quote can create project directly with prefilled values.
- Project is linked back to quote automatically.
- Job creation uses linked project and quote.
- Invoice created after project exists inherits project automatically.
- User understands clear sequence:
  Accepted Quote -> Project -> Job -> Invoice(s)
