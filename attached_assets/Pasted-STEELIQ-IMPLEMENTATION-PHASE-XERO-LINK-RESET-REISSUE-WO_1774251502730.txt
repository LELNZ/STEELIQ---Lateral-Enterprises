CURRENT PROBLEM TO FIX

A real workflow gap exists:

- An invoice was pushed to Xero as a test
- The Xero invoice was then deleted/voided in Xero
- SteelIQ successfully reads back the Xero status (e.g. DELETED)
- The user can return the invoice to draft and mark ready again
- But push-to-Xero still refuses because the invoice still has existing Xero linkage fields and SteelIQ correctly prevents duplicate re-push over an already linked invoice

This leaves the user stuck.

CURRENT TRUTH / DESIGN INTENT

The SteelIQ invoice should remain the internal commercial/control record.
Deleting or voiding the Xero-side draft/test invoice should NOT force the user to delete the SteelIQ invoice and create a brand-new SteelIQ invoice.

The correct model is:
- same SteelIQ invoice record remains
- prior Xero linkage can be explicitly reset/reissued through a controlled workflow
- next push creates a new Xero invoice record with a new Xero id/number
- audit trail is preserved

PRIMARY OBJECTIVE

Implement a controlled Xero link reset / reissue workflow so the same SteelIQ invoice can be re-pushed to Xero after the prior Xero invoice has been deleted or voided in Xero, without breaking auditability and without allowing accidental duplicate re-push for still-active Xero-linked invoices.

This phase should establish:
1. clear rules for when a Xero link may be reset
2. explicit user confirmation workflow
3. preserved audit trail of prior Xero linkage
4. invoice detail UI guidance for deleted/voided Xero states
5. safe ability to push the same SteelIQ invoice again after reset
6. no weakening of existing duplicate-push protections for normal cases
