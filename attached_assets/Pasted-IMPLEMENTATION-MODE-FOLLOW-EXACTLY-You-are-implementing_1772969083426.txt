PART A — PHOTO PERSISTENCE FIX
--------------------------------------------------

T031 — Persist customer photo binaries in PostgreSQL instead of volatile filesystem

Problem:
Photo metadata persists in DB, but photo files stored in uploads/item-photos/ do not survive reliably in Replit.
Result:
- drawings survive because they are regenerated
- photos disappear because they are stored once and lost from filesystem
- quotes/previews/PDFs then show broken/missing images

Required implementation:

1. Add the minimum required persistent storage for uploaded customer/item photos in PostgreSQL.
   Preferred implementation:
   - separate table for item photo binaries
   Suggested columns:
   • key (primary key / stable lookup key)
   • data (binary / bytea)
   • mimeType
   • createdAt
   Optional only if truly needed:
   • sizeBytes

2. Keep existing photo metadata/key usage as stable as possible.
   The QuoteDocument / Snapshot / Renderer pipeline should continue referring to photo keys, not be redesigned.

3. Update the photo upload flow so uploaded photos are persisted to PostgreSQL-backed storage.
   Filesystem storage may be removed or left as non-primary fallback, but DB must be the source of truth.

4. Update the photo serving route to read from DB-backed storage.

5. Update any delete/cleanup logic so DB-backed photo blobs are deleted only when appropriate and not while still referenced.

Acceptance criteria:
- newly uploaded customer/item photos persist across Replit restarts/rebuilds
- quote preview can still render photos
- PDF export can still embed photos
- no quote lifecycle redesign is introduced


T032 — Preserve compatibility with existing quote/photo references

Problem:
Existing quote/snapshot/photo references already depend on photo keys.

Required implementation:

1. Do NOT redesign the photo reference model if avoidable.
2. Existing metadata structures should continue to work with photo keys.
3. Migrate serving/storage behind the existing key abstraction.

Acceptance criteria:
- preview and PDF continue to resolve photos by key
- minimal ripple effects through quote pipeline


T033 — Backfill / transition behavior

Problem:
Some existing photo keys refer to missing files on disk.

Required implementation:

1. Add the smallest safe transition handling.
2. If an old photo file still exists on disk and is not yet in DB storage, allow a transition path if practical.
3. If a file is already gone, fail gracefully.
4. Do NOT over-engineer a historical recovery system.

Acceptance criteria:
- system handles mixed legacy state safely
- no crashes on old broken photo references
- new uploads use persistent storage going forward
