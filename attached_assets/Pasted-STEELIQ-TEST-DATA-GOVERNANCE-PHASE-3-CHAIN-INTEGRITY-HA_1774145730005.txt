PHASE A — CUSTOMER / CONTACT TEST-DATA SAFETY

OBJECTIVE

Allow safe governance handling for test/demo customers and contacts where they are genuinely isolated, while protecting any CRM record that is shared with live operational data.

REQUIRED IMPLEMENTATION

1. Review the current customer and contact model and identify the safest architecture-preserving way to support governance for CRM records.

2. Governance behavior for customers/contacts must follow these rules:
- only Owner/Admin can perform governance actions
- only records explicitly classified as test/demo may be eligible
- if a customer/contact is linked only to test/demo chains and not shared with live records, governance archive/delete may be allowed subject to safeguards
- if a customer/contact is linked to any live/non-test operational or commercial records, delete must be blocked
- if shared with live/non-test records, archive must also be blocked unless there is an architecture-safe reason it can be hidden without harming live operations
- the system must clearly explain why the CRM record is protected when blocked

3. At minimum, implement:
- governance review visibility for eligible test/demo customers/contacts if safe to add in this phase
- downstream/shared-usage detection
- delete blocking for shared live CRM records
- archive/delete only where isolation rules pass

4. If full CRM governance actions cannot be safely completed without architecture drift:
- implement detection + visibility + blocking first
- do not force destructive actions
- report what remains for a later phase

ACCEPTANCE CRITERIA

- Shared live CRM records are protected
- Isolated test/demo CRM records are either safely governable or explicitly detected and reported
- No live customer/contact record can be accidentally deleted because it was touched by test data
