OBJECTIVE

Replace snapshot-style PDF export with a true document PDF engine that generates customer quote PDFs from the quote data/render model itself.

Current problem:

- Export is effectively a snapshot of the preview DOM

This pass is the full engine pass for customer quote PDF export.


--------------------------------------------------
T025 — Remove snapshot-based export approach
--------------------------------------------------

Problem:
The current export path depends on preview DOM capture (html2canvas/jsPDF image slicing), which is not acceptable.

Required implementation:

1. Remove the current DOM snapshot export path as the primary export method.
2. Do NOT rely on:
   - html2canvas page screenshots
   - rasterized full-page preview capture
   - opening a new preview window purely to auto-export screenshots
3. Keep any fallback only if absolutely necessary, but the main export path must be true document generation.

Acceptance criteria:
- PDF export no longer depends on screenshotting the preview page
- Export does not require a temporary preview browser tab just to generate output


--------------------------------------------------
T026 — Build a true PDF document engine from QuoteDocument / QuoteRenderer output
--------------------------------------------------

Problem:
PDF export must be generated from structured quote data, not from page pixels.

Required implementation:

1. Build the PDF export from the existing architecture:
   - Quote revision snapshot
   - QuoteDocumentModel
   - QuoteRenderModel / renderer output
   - dedicated PDF layout/export layer

2. Reuse the existing renderer/data structures as much as safely possible.

3. If needed, introduce a small PDF-specific adapter layer, for example:
   - QuoteRenderModel -> PdfRenderModel
   but keep it small and explicit.

4. The PDF engine must produce:
   - true laid-out PDF pages
   - controlled sections
   - controlled pagination
   - repeatable output independent of browser print quirks

Acceptance criteria:
- PDF export is generated from structured document data
- No screenshot-based rendering is used as the main export
- Output is meaningfully closer to a real document engine


--------------------------------------------------
T027 — Generate controlled A4 pages with real pagination
--------------------------------------------------

Problem:
Customer PDFs need proper page construction, not browser-controlled or image-sliced output.

Required implementation:

1. Implement true A4 portrait page generation.
2. Apply controlled page margins and usable content width.
3. Support multi-page documents with explicit pagination logic.
4. Prevent awkward splitting of critical sections where practical.
5. Keep content readable and professionally scaled.

Required content support:
- header / branding
- customer and project details
- quote number / revision / dates
- totals summary
- exclusions / terms / payment blocks
- schedule of items
- drawings
- included site photos where applicable

Acceptance criteria:
- PDF pages are generated as true A4 pages
- Multi-page output is controlled by the engine, not by browser print
- Content is readable and appropriately scaled
