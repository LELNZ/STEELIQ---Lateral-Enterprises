CURRENT BUSINESS NEED

Subcontractor pricing methods differ. The current system needs to support real pricing styles such as:

- Timber window / door removal: priced per m²
- Sealant & weatherproofing: priced per linear metre
- Flashing / finish work: often priced per linear metre
- Aluminium window installation: priced per m²
- Aluminium façade install: distinct install class with its own inclusions/exclusions and likely per m² pricing

The system should not force all install/removal pricing into one method.

PRIMARY OBJECTIVE

Add pricing basis support to the install/removal library and executive summary calculation flow so labour/scope components can be priced using:
- per item
- per m²
- per l/m

with automatic quantity derivation where possible, especially for l/m-based rows.

PHASE C — MEASUREMENT RULES

Implement automatic quantity derivation for each basis.

Required behavior:
1. per_item
- quantity derived from item count / row quantity as current architecture truth dictates

2. per_m2
- quantity derived from item area in m²

3. per_lm
- quantity derived automatically from item dimensions using a clear default rule:
  perimeter lineal metres = 2 × (width + height) / 1000

This default rule must be stated clearly in the UI and/or code/report.

Do not require users to manually calculate l/m in the normal path.

2. Sell should continue to follow the existing markup/override model already hardened in prior phases.
3. Card totals and overall summary totals must remain consistent.
4. Per-line display should show enough information for a user to understand:
- basis used
- quantity used
- rate used
- resulting line total

Do not break existing enable/disable row behavior.
