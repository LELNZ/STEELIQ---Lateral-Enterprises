T005 — Improve First-Item Speed — Eager Job Creation

Good idea, but high risk. Needs more caution.

This is the one task I would soften.

Your hypothesis is probably correct:

first item delay likely includes job creation

But the proposed fix:

auto-create a job after 1 second when job name is entered

can create side effects:

lots of accidental draft jobs

half-created records

unexpected autosaves

confusing state if the user changes the job name or backs out

Safer alternative

I’d recommend this instead:

Revised T005

Investigate first-item delay first, then implement the lowest-risk performance fix.

Prefer this order:

confirm the actual cause of the 6–8 second delay

if job creation is the bottleneck, pre-create only when minimum required estimate fields are present, such as:

jobName

address or siteType

only create once

keep it clearly in draft state

do not create empty or near-empty jobs

Better trigger than only jobName

Just job name alone is too early.

A safer trigger is something like:

job name non-empty

site type selected

user has interacted with item creation flow

Or:

create the job when user first enters Config meaningfully / first interacts with item setup

Recommendation

Approve the idea, but ask Replit to measure before implementing, and avoid creating drafts too aggressively.
