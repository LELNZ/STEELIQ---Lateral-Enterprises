This must change.

That violates the original constraint:

“PDF generation integration using the existing PDF engine (no changes to PDF engine internals)”

“Treat PDF engine as a black box”

Also, building a second PDF generator in the client creates long-term divergence:

quote preview ≠ pdf output

template rules duplicated in TS

font embedding, pagination, headers/footers become brittle

hard to guarantee historical revision fidelity

Correct T011 approach (what you should instruct Replit to do instead)

Locate the existing quote PDF engine path

Find the server module/route/service already used for quote PDFs (even if currently unused).

Add a thin orchestration endpoint only if needed

Example pattern:

POST /api/quotes/:id/revisions/:revId/pdf (generate/store)

GET /api/quotes/:id/revisions/:revId/pdf (stream/download)

Keep it a wrapper that calls the existing engine; do not modify engine internals.
