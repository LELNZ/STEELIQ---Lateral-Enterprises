NOW IMPLEMENT THE FOLLOWING TASKS EXACTLY:

Objective:
Audit and fix quote preview item photo rendering so captions/lightbox do not appear for unresolved images, and confirm whether current storage is adequate.

Context:
In quote preview, an item can show:
- generated drawing
- site/customer photo
- photo caption
- lightbox/modal viewer

Observed issue:
A quote item shows “Photo unavailable” in preview, but still shows a photo caption beneath it.
Opening the image viewer/lightbox also results in a blank image with the caption still present.
This suggests metadata exists but the resolved image source is invalid or missing.

Required tasks:

1. Inspect the current photo storage and resolution path end-to-end.
   Determine:
   - where item photos are stored today
   - what fields are stored in the database or snapshot for photos
   - how those fields are transformed into QuoteDocumentModel
   - how QuoteRenderer / preview resolves the final image src
   - how the lightbox decides whether an image is openable

2. Identify the exact root cause.
   Examples of acceptable root-cause categories:
   - stale metadata but missing actual file
   - broken URL/path mapping
   - renderer emits photo block when no valid renderable src exists
   - preview caption/lightbox visibility checks are too loose
   - fallback placeholder and openable image states are incorrectly mixed

3. Fix the preview behavior with the smallest safe change.
   Requirements:
   - do not show clickable/lightbox-enabled photo UI unless there is a valid resolved image source
   - do not show photo caption beneath an unavailable/missing image unless this is explicitly intended and implemented consistently
   - keep graceful fallback behavior where appropriate
   - do not break drawings or valid photo rendering

4. Confirm whether a different database is actually needed.
   - If current storage is sufficient, explicitly say so
   - If storage is the root problem, explain exactly why
   - Do not redesign storage unless absolutely necessary

5. Keep the change focused.
   - no broad media-system rewrite
   - no schema redesign unless strictly required
   - no speculative upload subsystem changes

Acceptance criteria:
- valid photos still render correctly
- missing/broken photos do not show misleading caption/lightbox behavior
- the system clearly distinguishes between:
  - photo metadata exists
  - photo source resolves
  - photo is renderable/openable
- explanation is provided on whether current storage is adequate
