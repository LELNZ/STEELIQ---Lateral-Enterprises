Implement a focused UX / workflow pass before T018 QuoteRenderer architecture.

This pass must improve:
1. quote/revision structure and numbering clarity
2. quote media inclusion workflow
3. media viewer usability

Do NOT redesign the rendering architecture.
Do NOT bypass QuoteDocument.

Architecture constraints

Preserve the existing pipeline direction:

Quote Builder
-> QuoteDocument
-> Quote Preview
-> Quote Renderer

Business rules to preserve

TASK 1 — DEFINE AND IMPLEMENT QUOTE / REVISION STRUCTURE UX
==================================================

Goal

Make it clear how the system organizes:
- one estimate/project
- many quotes under that estimate
- many revisions under each quote

Required model rule

Use this hierarchy:

Estimate / Project
-> many Quotes
-> each Quote has many Revisions

Interpretation rules

- Revision = same commercial offer, updated details
- New Quote = alternate option / alternate scope / alternate package / alternate pricing path

Examples:
- same quote, changed dimensions or updated details = new revision
- standard option vs premium option = separate quote
- base scope vs optional extras = separate quote

Implementation requirements

A. Improve action labels in the estimate / exec summary workflow

Current labels are unclear.

Replace them with clearer actions such as:
- Update Existing Quote
  subtext: creates a new revision on the selected quote
- Create New Quote
  subtext: creates a separate quote record under this estimate/project

Use the actual UI patterns already in the codebase, but the wording must clearly communicate:
- revision on existing quote
- separate new quote under same estimate

B. Quote numbering and revision numbering

Implement/confirm:
- Quote number is auto-generated by system
- Revision number is auto-generated by system
- Revision number may be editable if already supported, but the system remains the default source of sequencing
- Do not break existing quote/revision persistence

C. Show quote family / source estimate relationship in UI

Add or improve display so that users can clearly see quotes belong to the same source estimate.

At minimum:
- In quote list rows/cards, show source estimate reference / project reference
- In estimate/project context, add or improve a Quote History / Related Quotes section showing:
  - quote number
  - current revision
  - status
  - updated date
  - whether it is current/alternate/archived if such status exists

Best effort if some pieces already exist:
- enhance rather than rebuild

Acceptance criteria

- Users can understand the difference between creating a new quote and creating a revision
- Multiple quotes from the same estimate are visibly tied to that estimate/project
- Quote and revision numbers remain system-generated and consistent
- No lifecycle regression introduced

TASK 2 — IMPROVE QUOTE MEDIA INCLUSION WORKFLOW
==================================================

Goal

Keep drawings included by default, and make site photo inclusion explicit and easy to manage at quote composition time.

Current problem

Photos only appear in quote preview when includeInCustomerPdf === true, but the control is too hidden in the photo/gallery workflow.

Required product behavior

- Drawings included by default
- Site photos excluded by default
- User can explicitly choose which photos to include in customer quote output
- This control must be available closer to quote composition / preview, not buried only in the gallery flow

Implementation requirements

A. Keep photo inclusion separate from Visible Specs

Do NOT add photo toggles to the Visible Specs panel.

Visible Specs is only for:
- configuration/spec fields
- dimensions
- glazing
- hardware
- pricing/spec display fields

Photos are media controls and must remain separate.

B. Add item-level media inclusion controls in Exec Summary and/or Quote Preview editing flow

For each quote item, add a small Media section or equivalent control area.

Recommended controls:
- Include drawing in quote (default ON)
- Include site photos in quote (default OFF if no photo individually selected)
- Per-photo include toggle / checkbox / selector for each item photo
- Show thumbnails for reference where practical

If the existing data model already stores photo-level includeInCustomerPdf, use that and do not redesign it.

If there is already a drawing inclusion assumption, preserve it unless there is a real supported toggle.

C. Final quote composition visibility

The estimator should be able to see, before finalizing:
- which items have customer-visible photos
- how many photos are included

At minimum:
- included photos are visually obvious in preview/edit flow
- user does not have to return to old hidden gallery workflow to control PDF inclusion

Acceptance criteria

- Drawings remain included by default
- Site photos are opt-in and selectable
- Photo inclusion controls are not hidden only inside the gallery flow
- Photo inclusion is not placed in Visible Specs
- Existing quote preview still renders only selected customer-facing photos

TASK 4 — QUOTE LAYOUT / PREVIEW UX RECOMMENDATIONS IMPLEMENTATION
==================================================

Goal

Reduce dependence on "show more / show less" for customer-facing quote content and make the item layout more printable/document-friendly.

Current issue

Interactive collapsing is not appropriate for printable/PDF output.
Long spec content can overwhelm the row/card layout.

Required direction

For the preview/editor layout:
- wrap long configuration text
- constrain spec columns so they do not become too wide
- allow the item block to grow vertically
- where appropriate, use a second specs column before resorting to collapse
- keep photos in a separate media block below drawing/specs

Recommended item layout

Top section:
- left: drawing
- right: key specs (wrapped, controlled width, printable-friendly)

Below:
- optional site photos row/gallery

So conceptually:

[ Drawing ] [ Key Specs ]

[ Optional site photos ]

Implementation requirements

A. Review current item layout in quote preview
B. Improve long-text behavior for spec values such as Configuration
C. Reduce reliance on chevron-based show more / show less in document-facing layout
D. Keep the output printable-friendly and closer to final document behavior

Do not fully redesign the renderer in this task.
This is a preview/layout cleanup pass only.

Acceptance criteria

- Long spec text wraps cleanly
- Spec columns remain readable
- Preview layout is more document-friendly
- Photos live in a separate media block, not mixed into spec visibility controls
- The preview does not rely on collapsed interactive content for core item details

TASK 5 — INVESTIGATE AND IMPROVE INTERMITTENT BUILDER LAG
==================================================

Goal

Reduce intermittent lag in the quote builder flow, especially after entering names and then scrolling long builder content on mobile/site-visit use.

Current issue

The app feels intermittently laggy when moving through the builder page, especially on long/complex pages.

Implementation requirements

A. Profile or inspect likely causes in the builder page

Look for likely causes such as:
- whole-page rerenders after small field updates
- expensive derived calculations on every keystroke
- image/drawing thumbnail rerenders
- oversized state updates high in the component tree
- heavy hidden components mounted unnecessarily

B. Apply focused performance improvements without redesign

Examples of acceptable fixes:
- memoize heavy subcomponents
- reduce unnecessary rerenders
- debounce non-critical recalculations
- lazy render heavy media where practical
- avoid rerendering large item lists when a single field changes
- ensure hidden/offscreen helper rendering is not mounted in the builder if not needed there

C. Prioritize mobile responsiveness

This app is used onsite on phones, so prefer improvements that help mobile browsing and long-form editing.

Acceptance criteria

- Builder feels more responsive in common editing/scrolling flow
- No functional regressions
- Changes are focused and not a broad architecture rewrite
