==================================================
WORKSTREAM A — COMMERCIAL BILLING RULES LOCK
==================================================

OBJECTIVE
Make invoiceable value and invoice-type behavior explicit, safe, and operationally coherent.

Required business rules to implement and/or make explicit in code + UI:

1. Base contract value
- Accepted quote value is the base contract value.

2. Deposit invoices
- Deposit invoices draw down only against the allowed deposit allocation.
- Existing deposit cap logic remains in force.
- UI must clearly show:
  - base contract value
  - deposit allowance
  - deposit already invoiced
  - deposit remaining

3. Progress invoices
- Progress invoices invoice against the remaining uninvoiced base contract value.
- They do not increase contract value.

4. Final invoice
- Final invoice should represent the remaining balance:
  base contract value
  + approved variation total
  - already invoiced
- It should auto-suggest the remaining amount.

5. Retention
- Retention must be explicitly modeled as withheld contract value, not an accidental leftover.
- Required behavior:
  - support retention percentage or amount on the accepted commercial scope if feasible in current model
  - retention release invoice can only draw from retention held
- If full retention implementation is too large for this phase, implement the model groundwork + UI placeholders and explain the exact next step.

6. Over-invoicing protection
- The system must prevent:
  - invoicing above base accepted value + approved variations
  - deposit invoicing above deposit allowance
  - retention release above retained amount

7. Status/locking rules
- Once invoice is pushed to Xero, commercial edits affecting financial meaning must be controlled and warned.

OBJECTIVE

Required behavior:

1. Add variation record model

- reason / description
- amountExclGst
- status

Recommended statuses:
- draft
- sent
- approved
- declined
- invoiced

2. Approved variation total
- Only approved variations increase total invoiceable value.
- Unapproved variations must not affect invoice allowance.

3. Variation UI surfaces
At minimum provide operational surfaces on:
- Project detail page
- Job detail page if job-linked
- possibly Quote detail if source-quote-linked context is useful

Required UI actions:
- create variation
- edit draft variation
- mark variation approved / declined
- create invoice from approved variation

4. Variation invoice creation
- Variation invoice must be created from an approved variation record
- It must inherit:
  - customer
  - project
  - source quote
  - source job if linked
  - source variation
- Variation invoice amount must not exceed approved variation value remaining to invoice

5. Commercial visibility
On relevant pages show:
- total approved variation value
- total variation invoiced
- remaining variation value to invoice

6. Preserve existing invoice list / Xero flow
- Variation invoices should still use existing invoice/Xero pipeline
- Do not create a parallel finance/export system
